# Cosy Domain Provider — Frontend

React SPA for managing `cosy-hosting.net` subdomains. Backend: `../cosy-domain-provider` (Spring Boot).

## Commands

```bash
bun dev            # Dev server at localhost:5173 (proxies /api → http://127.0.0.1:8080)
bun run build      # TypeScript check + Vite production build (bun build conflicts with bun's bundler)
bun lint       # Biome check
bun lint:fix   # Biome auto-fix
bun gen:api    # Regenerate API client from running backend OpenAPI spec (backend must be up)
```

## Key conventions

- **Styling: Tailwind utility classes first.** Prefer `className="..."` over `style={{...}}`. Only use inline styles for values that cannot be expressed as Tailwind classes (oklch/gradient/textShadow/boxShadow literals, dynamic numeric values with no Tailwind equivalent, or CSS properties with no utility class). See `docs/patterns/styling.md`.
- **Tailwind design tokens:** The design system's CSS variables are registered as Tailwind tokens. Use named utilities like `bg-background`, `text-foreground`, `border-destructive`, `rounded-radius-sm` — never the raw `[var(--x)]` arbitrary syntax.
- **File naming:** Component files use **kebab-case** (e.g. `user-menu.tsx`, `billing-header.tsx`). Logic hook files use **camelCase** (e.g. `useDashboardLogic.ts`, `useDropdown.ts`). Each component folder also has an optional `lib.ts` for pure utility functions, constants, and types — only create it when there's meaningful pure logic to extract (not just 1-2 trivial lines).
- **`lib.ts` pattern:** Pure functions, constants, and types that don't need React context live in a co-located `lib.ts` file (e.g. `src/pages/domain-detail/lib.ts`). The hook imports from `./lib`; other components import from the authoritative path. See `docs/patterns/component-structure.md`.
- **Logic hooks are mandatory for non-trivial components:** Any component with state, derived values, or async handlers must extract that logic into a co-located `useMyComponentLogic.ts`. The component file itself should be thin JSX only. See `docs/patterns/component-structure.md`.
- **Validators belong in `src/lib/validators.ts`:** All field validation functions (`isValidUsername`, `isValidPassword`, `isValidEmail`, etc.) live there. Never define them inline in a component or hook, and never put them in a component-level `lib.ts`.
- **`FormField` component:** All labeled inputs use `<FormField>` (`src/components/ui/form-field.tsx`). Use `suffix` for a static text overlay (e.g. `suffix=".play.cosy-hosting.net"`) and `endDecorator` for a ReactNode slot (e.g. a password eye-toggle button). Do not use raw `<input>` + `<label>` pairs. See `docs/patterns/form-inputs.md`.
- **`Icon` component:** Use `<Icon src={...} />` (`src/components/ui/icon.tsx`) for `.webp` pixel icons — it applies CSS masking so the icon fills with `currentColor`. Never use `<img>` for icons that need to match text/foreground color.
- **Barrel exports:** Every component subfolder has an `index.ts` that re-exports its public API. Pages use `index.tsx` directly as their export point (no separate barrel needed).
- **Linting/formatting: Biome only.** No ESLint, no Prettier. Run `npm run lint:fix` before committing.
- **Never edit generated files:** `src/routeTree.gen.ts` (auto-generated by TanStack Router on dev start) and `src/api/generated/` (generated by Orval from OpenAPI spec).
- **i18n:** Translation keys live in `src/i18n/resources.ts` as TypeScript (not JSON). Always add keys to **both** `en` and `de`.
- **API client:** Custom logic goes in `src/hooks/useDataLoading` and `src/hooks/useDataInteractions`, not in the generated API code.
- **New routes:** Create the page in `src/pages/`, create the route file in `src/routes/`. The route tree regenerates automatically on next dev server start.
- **Testing:** Use `data-testid` attributes for Playwright selectors (`page.getByTestId(...)`).

## Keeping docs up to date

`CLAUDE.md` and `docs/patterns/` are living documents. When a refactor introduces or solidifies a new convention, update them in the same session — don't leave the docs trailing the code.

**Update when:**
- A new structural pattern emerges (e.g. a new file type, a new folder convention, a new hook pattern)
- An existing rule is refined or an exception is established
- A design principle is applied consistently for the first time (e.g. the first time a pattern is used in 2+ places, it's worth documenting)

**Don't update for:**
- One-off implementation decisions that aren't meant to generalise
- Changes already covered by existing documentation
- Minor refactors with no architectural significance

If the pattern warrants a new `docs/patterns/` entry, create the file. If it fits an existing one, extend it. Keep entries concise — a rule, why it exists, and a short example.
